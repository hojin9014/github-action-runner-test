# 2026-06-08 작업 정리

## 1. NICE 본인인증 연동 방식 정리

NICE PASS 본인인증은 사용자가 PASS 앱을 쓰는지 여부와 별개로, 서버가 NICE와 어떤 방식으로 연동하는지에 따라 구분된다.

검토한 방식은 다음 두 가지다.

- SiteCode + SitePassword 방식
- NICE API 방식

SiteCode + SitePassword 방식은 기존 CheckPlus 계열에서 많이 쓰는 방식이다. 서버가 NICE에서 발급받은 `SiteCode`, `SitePassword`를 사용해 요청 데이터를 암호화하고, 인증 완료 후 NICE callback으로 받은 암호화 데이터를 복호화한다.

흐름은 다음과 같다.

```text
사용자 본인인증 버튼 클릭
-> Controller 요청
-> NiceService.getEncData()
-> NICE 요청용 plainData 생성
-> CPClient.fnEncode()
-> encData를 JSP에 전달
-> NICE 인증창 호출
-> 인증 완료
-> callback URL로 암호화 결과 수신
-> NiceService.getDecode()
-> CPClient.fnDecode()
-> 이름, 생년월일, 휴대폰번호, CI/DI 등 복호화
```

NICE API 방식은 REST API 기반으로, 앱 등록과 승인 후 Secret Key 또는 API 인증 정보를 사용해 연동하는 방식이다. 신규 서비스라면 API 방식이 더 현대적인 구조에 맞지만, 계약/승인 없이는 실제 API 테스트가 어렵다.

계약 전 개발은 Mock 구현체로 진행할 수 있다고 정리했다. Mock 구현체는 NICE 서버를 실제 호출하지 않고, 인증 성공처럼 가짜 응답을 반환하는 대체 구현체다.

## 2. Spring과 Spring Boot 차이 정리

Spring은 DI, IoC, MVC, 트랜잭션 같은 기능을 제공하는 프레임워크다.

Spring Boot는 Spring을 기반으로 자동 설정, 내장 서버, 간편한 실행 구조를 제공한다.

정리하면 다음과 같다.

```text
Spring = 직접 설정이 많은 프레임워크
Spring Boot = Spring을 빠르게 실행하고 배포하기 쉽게 만든 프레임워크
```

이번 프로젝트는 Spring Boot 기반으로 진행하기로 했다.

## 5. IntelliJ 관련 문제와 해결 방향

IntelliJ에서 다음 오류가 발생했다.

```text
Cannot run program ".../jdk1.8.0_102.../bin/java"
CreateProcess error=2
```

원인은 IntelliJ가 존재하지 않는 JDK 1.8 경로를 바라보고 있었기 때문이다.

현재 PC에서 확인된 JDK 후보는 다음이었다.

```text
C:\Users\splunk\.jdks\ms-17.0.18
C:\Users\splunk\.jdks\ms-21.0.10
C:\Program Files\Java\jdk-21.0.10
```

Spring Boot 3.x는 Java 17 이상이 필요하므로, Project SDK는 Java 17로 설정하는 것이 좋다.

추가로 `com.nicecheck.poc.NicePocApplication`에 접근할 수 없다는 IntelliJ 오류는 JDK 문제라기보다 Maven reload, source root, run configuration 문제일 가능성이 높다고 정리했다.

확인할 항목은 다음과 같다.

```text
pom.xml -> Add as Maven Project
src/main/java -> Sources Root
Run Configuration 재생성
Use classpath of module 확인
Invalidate Caches 후 재시작
```

## 8. OCI Tomcat 구조 확인

OCI 인스턴스에서 확인한 Tomcat 경로는 다음과 같다.

```text
CATALINA_HOME = /usr/share/tomcat10
CATALINA_BASE = /var/lib/tomcat10
설정 파일 = /etc/tomcat10
배포 경로 = /var/lib/tomcat10/webapps
서비스명 = tomcat10
```

`/usr/share/tomcat10/bin`에는 `startup.sh`, `shutdown.sh`, `catalina.sh` 등이 있다.

하지만 WAR 배포에 중요한 경로는 `bin`이 아니라 다음이다.

```text
/var/lib/tomcat10/webapps
```

현재 workflow는 `ROOT.war`로 배포하도록 작성되어 있다.

```text
target/nice-poc.war
-> /var/lib/tomcat10/webapps/ROOT.war
```

따라서 배포 성공 시 접속 주소는 다음이다.

```text
http://OCI_PUBLIC_IP:8080/
```

`nice-poc.war`로 배포하면 접속 주소는 다음이 된다.

```text
http://OCI_PUBLIC_IP:8080/nice-poc/
```

## 9. GitHub Actions self-hosted runner 정리

OCI 인스턴스에 GitHub Actions self-hosted runner를 설치했다.

runner가 GitHub Actions 화면에서 `Idle`로 보이는 것은 정상이다.

상태 의미는 다음과 같다.

```text
Idle = 온라인이며 job 대기 중
Busy = workflow 실행 중
Offline = 연결 끊김
```

수동 실행용 트리거는 다음이다.

```yaml
workflow_dispatch:
```

push 자동 실행용 트리거는 다음이다.

```yaml
push:
  branches:
    - main
```

현재 workflow는 둘 다 포함한다.

```yaml
on:
  push:
    branches:
      - main
  workflow_dispatch:
```

runner label은 현재 다음 조합을 요구한다.

```yaml
runs-on:
  - self-hosted
  - Linux
  - x64
  - oci
  - tomcat
```

이 label이 실제 runner에 모두 있어야 job이 실행된다.

## 10. Maven 관련 오류

GitHub Actions 실행 중 다음 오류가 발생했다.

```text
mvn: command not found
```

원인은 OCI runner 서버에 Java는 있었지만 Maven이 없었기 때문이다.

확인된 Java 버전은 다음이었다.

```text
openjdk version "17.0.19"
```

해결 방법은 OCI 인스턴스에 Maven을 설치하는 것이다.

```bash
sudo apt update
sudo apt install -y maven
mvn -version
```

## 11. sudo 권한 관련 오류

GitHub Actions에서 Tomcat 제어 단계에서 다음 오류가 발생했다.

```text
sudo: a terminal is required to read the password
sudo: a password is required
```

원인은 GitHub Actions runner가 비밀번호를 입력할 수 없는 비대화형 환경인데, `sudo`가 비밀번호를 요구했기 때문이다.

workflow는 `sudo -n` 방식으로 수정했다.

`sudo -n`의 의미는 다음과 같다.

```text
비밀번호를 묻지 말고, 권한이 없으면 즉시 실패하라
```

현재 workflow의 Tomcat 관련 명령은 다음 형태다.

```bash
sudo -n /usr/bin/systemctl stop "$TOMCAT_SERVICE"
sudo -n /usr/bin/rm -f "$TOMCAT_WEBAPPS/$DEPLOY_WAR"
sudo -n /usr/bin/rm -rf "$TOMCAT_WEBAPPS/ROOT"
sudo -n /usr/bin/cp "target/$APP_WAR" "$TOMCAT_WEBAPPS/$DEPLOY_WAR"
sudo -n /usr/bin/chown tomcat:tomcat "$TOMCAT_WEBAPPS/$DEPLOY_WAR" || true
sudo -n /usr/bin/systemctl start "$TOMCAT_SERVICE"
sudo -n /usr/bin/systemctl status "$TOMCAT_SERVICE" --no-pager
```

하지만 Actions에서 여전히 다음 오류가 발생했다.

```text
sudo -n /usr/bin/systemctl start "$TOMCAT_SERVICE"
sudo: a password is required
```

이는 `hojin9014` 사용자에게 `systemctl start tomcat10` NOPASSWD 권한이 완전히 적용되지 않았거나, OCI 터미널에서 테스트할 때 sudo 인증 캐시가 남아 있었을 가능성이 있다.

검증할 때는 반드시 sudo 캐시를 지운 뒤 테스트해야 한다.

```bash
sudo -k
sudo -n /usr/bin/systemctl stop tomcat10
sudo -n /usr/bin/systemctl start tomcat10
sudo -n /usr/bin/systemctl status tomcat10 --no-pager
```

위 세 명령이 모두 비밀번호 없이 성공해야 GitHub Actions에서도 성공한다.

## 12. 현재 deploy.yml 요약

현재 배포 workflow의 목적은 다음이다.

```text
main push 또는 수동 실행
-> OCI runner에서 실행
-> Maven으로 WAR 빌드
-> Tomcat10 중지
-> ROOT.war로 배포
-> Tomcat10 시작
```

현재 환경 변수는 다음이다.

```yaml
APP_WAR: nice-poc.war
DEPLOY_WAR: ROOT.war
TOMCAT_SERVICE: tomcat10
TOMCAT_WEBAPPS: /var/lib/tomcat10/webapps
```

## 13. 남은 이슈와 다음 선택지

남은 핵심 이슈는 Tomcat 시작 단계의 sudo 권한이다.

선택지는 두 가지다.

### 선택지 A: sudoers를 정확히 수정

`hojin9014`에게 Tomcat 서비스 제어와 배포 명령만 비밀번호 없이 허용한다.

검증은 반드시 다음처럼 한다.

```bash
sudo -k
sudo -n /usr/bin/systemctl start tomcat10
```

### 선택지 B: 재시작 없이 webapps에 WAR만 배포

Tomcat의 auto deploy를 사용해 `webapps`에 WAR만 복사한다.

이 경우 `hojin9014`에게 `/var/lib/tomcat10/webapps` 쓰기 권한을 주면 workflow에서 `sudo`를 줄일 수 있다.

예상 접속 방식은 두 가지다.

```text
ROOT.war 배포 -> http://OCI_PUBLIC_IP:8080/
nice-poc.war 배포 -> http://OCI_PUBLIC_IP:8080/nice-poc/
```
