# 2026-06-08 작업 정리 - GCP 기준

## 1. 진행 기준 정리

운영 환경은 GCP VM 기준으로 진행한다.

현재 목표 흐름은 다음과 같다.

```text
로컬 PC에서 코드 수정
-> GitHub main 브랜치로 push
-> GitHub Actions workflow 실행
-> GCP VM의 self-hosted runner가 job 수행
-> Maven으로 WAR 빌드
-> 같은 GCP VM의 Tomcat10 webapps에 배포
-> Tomcat에서 JSP 페이지 서비스
```

초기 운영 구성은 WEB/WAS를 분리하지 않고 GCP VM 한 대에 Tomcat과 GitHub Actions runner를 함께 두는 방식으로 시작한다.

## 2. NICE 본인인증 정리

검토한 NICE PASS 연동 방식은 두 가지다.

- SiteCode + SitePassword 방식
- NICE API 방식

SiteCode + SitePassword 방식은 NICE에서 발급받은 `SiteCode`, `SitePassword`를 사용해 서버에서 요청 데이터를 암호화하고, 인증 완료 후 callback으로 받은 암호화 데이터를 다시 복호화하는 구조다.

흐름은 다음과 같다.

```text
사용자 본인인증 버튼 클릭
-> Controller 요청
-> NiceService.getEncData()
-> NICE 요청용 plainData 생성
-> CPClient.fnEncode()
-> encData를 JSP에 전달
-> NICE 인증창 호출
-> 인증 완료
-> callback URL로 암호화 결과 수신
-> NiceService.getDecode()
-> CPClient.fnDecode()
-> 이름, 생년월일, 휴대폰번호, CI/DI 등 복호화
```

NICE API 방식은 REST API 기반으로 연동하는 방식이며, 계약과 승인 후 API 인증 정보를 받아야 실제 연동이 가능하다.

계약 전 개발은 mock 구현체로 진행할 수 있다. mock 구현체는 NICE 서버를 실제 호출하지 않고, 인증 성공처럼 보이는 가짜 응답을 반환하는 대체 구현체다.

## 3. Spring과 Spring Boot 차이

Spring은 DI, IoC, MVC, 트랜잭션 같은 기능을 제공하는 프레임워크다.

Spring Boot는 Spring을 더 빠르게 실행하고 배포할 수 있게 자동 설정, 내장 서버, 실행 구조를 제공한다.

정리하면 다음과 같다.

```text
Spring = 직접 설정이 많은 프레임워크
Spring Boot = Spring을 빠르게 실행하고 배포하기 쉽게 만든 프레임워크
```

현재 프로젝트는 Spring Boot 기반으로 진행한다.

## 4. 현재 프로젝트 상태

기존 Vue starter와 CRUD 구현은 제거했고, 현재는 가장 단순한 JSP 기반 Spring Boot 프로젝트로 구성했다.

현재 프론트는 Vue나 React가 아니라 servlet, JSP 방식이다.

현재 화면은 두 개다.

```text
/      -> 버튼이 있는 첫 페이지
/next  -> "버튼이 작동합니다" 문구를 보여주는 페이지
```

주요 파일은 다음과 같다.

```text
pom.xml
README.md
src/main/java/com/nicecheck/poc/NicePocApplication.java
src/main/java/com/nicecheck/poc/web/PageController.java
src/main/resources/application.properties
src/main/webapp/WEB-INF/views/index.jsp
src/main/webapp/WEB-INF/views/next.jsp
.github/workflows/deploy.yml
```

`NicePocApplication`은 Spring Boot 진입점이며, 외부 Tomcat WAR 배포를 위해 `SpringBootServletInitializer`를 상속한다.

`PageController`는 `/`, `/next` 요청을 JSP view로 연결한다.

`application.properties`에는 JSP view resolver와 로컬 실행 포트가 들어 있다.

```properties
server.port=8080
spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp
```

빌드 결과물은 다음 경로에 생성된다.

```text
target/nice-poc.war
```

## 5. IntelliJ 관련 정리

IntelliJ에서 다음 오류가 발생했었다.

```text
Cannot run program ".../jdk1.8.0_102.../bin/java"
CreateProcess error=2
```

원인은 IntelliJ가 존재하지 않는 JDK 1.8 경로를 바라보고 있었기 때문이다.

Spring Boot 3.x는 Java 17 이상이 필요하므로 Project SDK는 Java 17로 설정하는 것이 적절하다.

확인할 항목은 다음과 같다.

```text
Project SDK = Java 17
pom.xml -> Add as Maven Project
src/main/java -> Sources Root
Run Configuration 재생성
Use classpath of module 확인
Invalidate Caches 후 재시작
```

## 6. Git과 .gitignore 정리

Git에는 소스코드와 빌드 설정을 올리고, 빌드 결과물은 올리지 않는 것이 일반적이다.

Git에 올리는 것:

```text
src/
pom.xml
README.md
.github/workflows/*.yml
.gitignore
기본 application.properties
```

Git에 올리지 않는 것:

```text
target/
build/
*.jar
*.war
*.class
.idea/
*.log
실제 비밀번호와 Secret이 들어간 property
```

빌드 결과물은 GitHub Actions runner가 매번 새로 생성한다.

## 7. GCP 운영 구성

초기 GCP 구성은 다음과 같다.

```text
GCP Compute Engine VM 1대
Ubuntu
Java 17
Maven
Tomcat 10
GitHub Actions self-hosted runner
```

접속 및 방화벽 기준은 다음과 같다.

```text
SSH: 22
Tomcat HTTP: 8080
```

GCP 방화벽에서 8080 포트가 열려 있어야 외부 브라우저에서 접속할 수 있다.

ROOT.war로 배포하면 접속 주소는 다음과 같다.

```text
http://GCP_PUBLIC_IP:8080/
```

`nice-poc.war` 이름 그대로 배포하면 접속 주소는 다음과 같다.

```text
http://GCP_PUBLIC_IP:8080/nice-poc/
```

현재 workflow는 `ROOT.war`로 배포하도록 구성되어 있으므로 기본 접속 주소는 `/`이다.

## 8. Ubuntu Tomcat10 경로

Ubuntu 패키지로 Tomcat10을 설치한 경우 일반적으로 다음 구조를 사용한다.

```text
CATALINA_HOME = /usr/share/tomcat10
CATALINA_BASE = /var/lib/tomcat10
설정 파일 = /etc/tomcat10
배포 경로 = /var/lib/tomcat10/webapps
서비스명 = tomcat10
```

`/usr/share/tomcat10/bin`은 Tomcat 실행 스크립트가 있는 위치다.

WAR 배포에 중요한 위치는 다음 경로다.

```text
/var/lib/tomcat10/webapps
```

현재 workflow의 배포 흐름은 다음과 같다.

```text
target/nice-poc.war
-> /var/lib/tomcat10/webapps/ROOT.war
```

## 9. GitHub Actions self-hosted runner

GCP VM에 GitHub Actions self-hosted runner를 설치해서 배포 job을 수행한다.

runner 상태 의미는 다음과 같다.

```text
Idle = 온라인이며 job 대기 중
Busy = workflow 실행 중
Offline = 연결 끊김
```

수동 실행 트리거는 다음 설정이다.

```yaml
workflow_dispatch:
```

push 자동 실행 트리거는 다음 설정이다.

```yaml
push:
  branches:
    - main
```

현재 workflow는 둘 다 포함한다.

```yaml
on:
  push:
    branches:
      - main
  workflow_dispatch:
```

현재 workflow는 다음 runner label 조합을 요구한다.

```yaml
runs-on:
  - self-hosted
  - Linux
  - x64
  - gcp
  - tomcat
```

GCP VM에 설치한 runner에도 `gcp`, `tomcat` label이 있어야 job이 잡힌다.

label을 아직 맞추지 않았다면 GitHub repository의 runner 설정에서 label을 추가하거나, workflow의 `runs-on` 조건을 runner label에 맞게 수정해야 한다.

## 10. Maven 설치 이슈

Actions 실행 중 다음 오류가 발생했었다.

```text
mvn: command not found
```

원인은 runner 서버에 Java는 있지만 Maven이 설치되어 있지 않았기 때문이다.

GCP VM에서 다음 명령으로 Maven을 설치해야 한다.

```bash
sudo apt update
sudo apt install -y maven
mvn -version
```

## 11. sudo 권한 이슈

GitHub Actions에서 Tomcat 제어 단계에 다음 오류가 발생했었다.

```text
sudo: a terminal is required to read the password
sudo: a password is required
```

원인은 GitHub Actions runner가 비밀번호를 입력할 수 없는 비대화형 환경인데, `sudo`가 비밀번호를 요구했기 때문이다.

workflow에서는 `sudo -n`을 사용한다.

`sudo -n`의 의미는 다음과 같다.

```text
비밀번호를 묻지 말고, 권한이 없으면 즉시 실패하라
```

GCP VM에서는 runner 실행 사용자에게 필요한 명령만 `NOPASSWD`로 허용해야 한다.

검증할 때는 sudo 인증 캐시를 지운 뒤 확인한다.

```bash
sudo -k
sudo -n /usr/bin/systemctl stop tomcat10
sudo -n /usr/bin/systemctl start tomcat10
sudo -n /usr/bin/systemctl status tomcat10 --no-pager
```

위 명령이 모두 비밀번호 없이 성공해야 GitHub Actions에서도 성공한다.

## 12. 현재 deploy.yml 요약

현재 workflow의 목적은 다음과 같다.

```text
main push 또는 수동 실행
-> GCP self-hosted runner에서 실행
-> Maven으로 WAR 빌드
-> Tomcat10 중지
-> ROOT.war로 배포
-> Tomcat10 시작
```

현재 환경 변수는 다음과 같다.

```yaml
APP_WAR: nice-poc.war
DEPLOY_WAR: ROOT.war
TOMCAT_SERVICE: tomcat10
TOMCAT_WEBAPPS: /var/lib/tomcat10/webapps
```

## 13. 남은 작업

다음 작업 순서는 다음과 같다.

```text
1. GCP VM에 Java 17, Maven, Tomcat10 설치 확인
2. GCP VM에 GitHub Actions self-hosted runner 설치 및 서비스 등록
3. runner label에 gcp, tomcat 추가
4. GCP 방화벽에서 8080 포트 허용
5. sudoers에서 runner 사용자에게 Tomcat 제어 권한 부여
6. GitHub Actions workflow 수동 실행
7. target/nice-poc.war가 ROOT.war로 배포되는지 확인
8. http://GCP_PUBLIC_IP:8080/ 접속 확인
```

현재 애플리케이션 목표 화면은 다음과 같다.

```text
GET /
-> 버튼 표시

버튼 클릭
-> GET /next
-> "버튼이 작동합니다" 표시
```
